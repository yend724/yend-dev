[Effect](https://effect.website/) のドキュメントを読んでいて、`Effect.fail` と `Effect.succeed` を三項演算子で返したときに `never` がきれいに消える挙動が気になったので、その仕組みを集合論の視点から整理したメモ。

`never` を**空集合**として解釈すると、ユニオンから消える・インターセクションを呑み込む・条件型のフィルタになる、といった挙動がすべて素直な集合演算の結果になる。和集合 `∪`、積集合 `∩`、部分集合 `⊂`、空集合 `∅` といった集合論の道具が、そのまま型の演算と対応する。

ただし Effect そのものを題材にするとライブラリとしての Effect の説明から入る必要があり、本筋から離れてしまう。本記事では同じ発想を持った最小限の型 `MyEffect<A, E>` を自前で作って話を進める。

## MyEffect での挙動

`MyEffect<A, E>` は「成功すれば `A` 型の値、失敗すれば `E` 型のエラー」を表す、いわゆる Result 相当の型である。タグ付きユニオンで素朴に定義する。

```ts
type MyEffect<A, E> =
  | { readonly _tag: "Success"; readonly value: A }
  | { readonly _tag: "Failure"; readonly error: E };

function succeed<A>(value: A): MyEffect<A, never> {
  return { _tag: "Success", value };
}

function fail<E>(error: E): MyEffect<never, E> {
  return { _tag: "Failure", error };
}
```

戻り値の型は次のようになっている。

- `succeed` は `MyEffect<A, never>`：成功しか作らないので、エラー型は `never`
- `fail` は `MyEffect<never, E>`：失敗しか作らないので、成功型は `never`

「ありえない側」を `never` で塞いでいる形である。

これを三項演算子で合成してみる。

```ts
const divide = (a: number, b: number) =>
  b === 0 ? fail(new Error("Cannot divide by zero")) : succeed(a / b);
```

`fail(...)` は `MyEffect<never, Error>`、`succeed(a / b)` は `MyEffect<number, never>` である。素朴に考えれば、三項演算子の戻り値はこれらのユニオンになるはずだ。

```ts
MyEffect<never, Error> | MyEffect<number, never>;
```

ところが、TypeScript の推論結果はこうなる。

```ts
// const divide: (a: number, b: number) => MyEffect<number, Error>
```

`never` が消えて、`MyEffect<number, Error>` にまとまっている。なぜこうなるのかを以降で説明する。

## 型を集合として見る

TypeScript の型は、**値を要素とする集合**として捉えることができる。

- `string` は「すべての文字列の集合」
- `number` は「すべての数値の集合」
- `boolean` は「`true` と `false` の2要素の集合」
- `"foo" | "bar"` は「`"foo"` と `"bar"` の2要素の集合」

ある値がある型を持つというのは、その値がその集合の要素であるということに対応する。型 `A` が型 `B` のサブタイプであるとは、集合 `A` が集合 `B` の部分集合(`A ⊂ B`)であるということと一致する。

この見方をすると、TypeScript の型演算子は集合演算と素直に対応する。

| TypeScript              | 集合演算 | 意味                              |
| ----------------------- | -------- | --------------------------------- |
| `A \| B`                | `A ∪ B`  | 和集合：A か B のどちらかに属する |
| `A & B`                 | `A ∩ B`  | 積集合：A にも B にも属する       |
| `A` が `B` のサブタイプ | `A ⊂ B`  | A は B の部分集合                 |

ここで「いちばん大きな集合」と「いちばん小さな集合」を考える。すべての値を含む集合と、何の値も含まない集合である。

- **トップ型**：すべての値を含む集合。TypeScript における `unknown` がこれにあたる。
- **ボトム型**：何の値も含まない集合、つまり**空集合**。これが `never` である。

TypeScript 公式の Handbook も `never` の振る舞いを次のように述べている。

> The `never` type is assignable to every type; however, no type is assignable to `never` (except `never` itself).
>
> — [TypeScript Documentation, "Narrowing"](https://www.typescriptlang.org/docs/handbook/2/narrowing.html#exhaustiveness-checking)

これを集合の言葉に翻訳すれば：

- 空集合 `∅` はすべての集合の部分集合である(`∅ ⊂ A` がどんな `A` についても成り立つ)→ どんな型にも代入できる
- 空集合に要素は存在しない → 何も `never` には代入できない

という当たり前の事実に対応している。

`unknown` と `never` は、集合の最上端と最下端としてきれいに対称をなす。`unknown` には何でも代入できるが、`unknown` を使うには絞り込み(narrowing)が必要だ。逆に `never` は絞り込まずにどこにでも使えるが、`never` を作る方法がない。

## 集合演算との対応

`never` を空集合と捉えると、いくつかの挙動が集合演算として素直に説明できる。

### ユニオン型から消える

```ts
type T = string | never; // string
type U = number | string | never; // number | string
```

これは集合の式 `A ∪ ∅ = A` に対応する。空集合と何かの和集合をとっても、空集合は何も足さない。だから `never` はユニオンから消える。

数の世界で `0` が加算の単位元(`x + 0 = x`)であるのと同じ役割を、`never` は型のユニオンに対して果たしている。

### インターセクション型ですべてを呑み込む

逆に、`never` をインターセクションに混ぜると結果は `never` になる。

```ts
type T = string & never; // never
```

これも `A ∩ ∅ = ∅` に対応する。空集合との積集合は空集合だ。

少し意外なのは、互いに重ならない型のインターセクションも `never` になることである。

```ts
type T = string & number; // never
type U = boolean & symbol; // never
```

「文字列でもあり数値でもある値」は存在しないから、その集合は空である。だから結果は `never` になる。**型が `never` に潰れたら、両立できない型を交差させているサインと読める**。

実務でよく出くわす例として、こんなインデックスシグネチャの状況がある。

```ts
interface A {
  foo: string;
}
interface B {
  foo: number;
}

type C = A & B;
// C は { foo: string & number } = { foo: never }
// foo に値を入れようとすると never に代入できなくてエラー
```

「`foo` に値を割り当てられない」というエラーは、「`string` と `number` を交差させた `foo` の集合は空だから `never` になっていて、空集合に要素を入れる方法はない」と読み解ける。

### 条件型でのフィルタリング

条件型と組み合わせると、`never` はユニオン型のフィルタとして機能する。

```ts
type KeepStrings<T> = T extends string ? T : never;

type R = KeepStrings<"a" | "b" | 0 | 1>;
// "a" | "b"
```

条件型はユニオンに対して分配されるので、内部的には次のような展開が起きている。

```ts
// ("a" extends string ? "a" : never)
// | ("b" extends string ? "b" : never)
// | (0   extends string ? 0   : never)
// | (1   extends string ? 1   : never)

// = "a" | "b" | never | never
// = "a" | "b"
```

「拒否したいメンバを `never` に潰してユニオンから蒸発させる」のが `never` を使ったフィルタの仕組みである。標準ライブラリの `Exclude<T, U>` も同じテクニックで作られている。

```ts
type Exclude<T, U> = T extends U ? never : T;
```

## MyEffect の推論結果の説明

ここまでの道具立てで、冒頭の `divide` の挙動を説明できる。

```ts
const divide = (a: number, b: number) =>
  b === 0 ? fail(new Error("Cannot divide by zero")) : succeed(a / b);
```

各ブランチの型はこうだった。

- `fail(new Error(...))` は `MyEffect<never, Error>`：成功する可能性が空集合
- `succeed(a / b)` は `MyEffect<number, never>`：失敗する可能性が空集合

三項演算子の戻り値型を決めるとき、TypeScript はまず両ブランチを共通の型コンストラクタ `MyEffect<_, _>` の適用と見なし、**各型パラメータの位置でユニオンをとる**。これは型の統一(unification)と呼ばれる挙動である。

```ts
// 1番目のパラメータ位置: never | number
// 2番目のパラメータ位置: Error | never

// よって、推論結果は
MyEffect<never | number, Error | never>;
```

ここで空集合の出番である。`A ∪ ∅ = A` の規則によって、

```ts
never | number = number
Error | never  = Error
```

が成り立つから、最終的な型は `MyEffect<number, Error>` になる。

TypeScript 5.9 で型レベルアサーションを書いて確かめると、両者が一致する。

```ts
type Equals<X, Y> =
  (<T>() => T extends X ? 1 : 2) extends <T>() => T extends Y ? 1 : 2
    ? true
    : false;

// true で型エラーが出ない = divide の戻り値は MyEffect<number, Error> と等価
const _check: Equals<ReturnType<typeof divide>, MyEffect<number, Error>> = true;
```

ちなみに、もし TypeScript が単純にタグ付きユニオンとして両ブランチをそのまま結合していたら、結果は次のようになるはずだった。

```ts
MyEffect<never, Error> | MyEffect<number, never>;
// = | { _tag: "Success"; value: never }   ← 居候
//   | { _tag: "Failure"; error: Error }
//   | { _tag: "Success"; value: number }
//   | { _tag: "Failure"; error: never }   ← 居候
```

「`value` が `never` の Success」「`error` が `never` の Failure」という、実体を作れない居候のメンバーが残る。実際には TypeScript は型統一によって各パラメータ位置でユニオンを計算するため、こうした居候は発生せず、`MyEffect<number, Error>` に集約される。

「**`fail` の成功値は空集合**」「**`succeed` のエラー値は空集合**」という事実を、TypeScript の型統一とユニオンの計算が吸収している。`MyEffect` という自前の型でも同じ挙動が再現できることから、これは Effect のために特別に組まれた挙動ではなく、空集合としての `never` の性質をそのまま利用したものだとわかる。

Effect の `Effect.succeed` / `Effect.fail` も、公式ドキュメントによれば同じ発想で型付けされている。

> The type of `success` is `Effect<number, never, never>`, which means:
>
> - It produces a value of type `number`.
> - It does not generate any errors (`never` indicates no errors).
> - It requires no additional data or dependencies (`never` indicates no requirements).
>
> — [Effect Documentation, "Creating Effects"](https://effect.website/docs/getting-started/creating-effects/#succeed)

「ありえない側を `never` で塞ぐ」という設計は、自前の `MyEffect` でも Effect でも共通する語彙である。

## 応用例

### 網羅性チェック

ユニオン型を `switch` で分岐させ、すべてのケースを処理し終えたあとに `never` 型として扱われる、というおなじみのパターンも、空集合の言葉で読み直せる。TypeScript Handbook はこの仕組みを次のように説明している。

> When narrowing, you can reduce the options of a union to a point where you have removed all possibilities and have nothing left. In those cases, TypeScript will use a `never` type to represent a state which shouldn't exist.
>
> — [TypeScript Documentation, "Narrowing"](https://www.typescriptlang.org/docs/handbook/2/narrowing.html#exhaustiveness-checking)

「すべての可能性を取り除いて何も残っていない状態」は、空集合そのものである。

```ts
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; side: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.side ** 2;
    default:
      const _exhaustive: never = shape; // ← ここで shape は never
      return _exhaustive;
  }
}
```

`switch` の各ケースは、`shape` が取りうる値の集合からそのケースに対応する部分集合を取り除いていく操作と読める。`circle` を処理するとそれが集合から消え、`square` を処理するとそれも消える。すべてのケースを書き切ったあとの `default` 節では、`shape` が取りうる値の集合は空、つまり `never` になっている。だからこそ `const _exhaustive: never = shape` という代入が通る。

新しいケース、たとえば `{ kind: "triangle"; ... }` を `Shape` に追加して `switch` を更新し忘れると、`default` 節での `shape` は `never` ではなく `Triangle` を含んだままになる。`Triangle` は `never` のサブタイプではないので、この代入はコンパイルエラーになり、ケース漏れに気づける。

### 「絶対に返らない」関数

例外を投げ続けたり、無限ループに入ったりする関数の戻り値を `never` で型付けるのも、同じ発想で説明できる。

```ts
function panic(message: string): never {
  throw new Error(message);
}
```

この関数が「正常に値を返す」というイベントの集合は空である。だから戻り値の型は空集合 = `never` になる。

これにより、TypeScript の制御フロー解析は `panic` を呼び出した時点で「そこから先は実行されない」と判断できる。

```ts
function getOrPanic(x: string | undefined): string {
  if (x === undefined) {
    panic("missing value"); // ← 戻ってこないので
  }
  return x; // ← ここで x は string
}
```

`x` の型は元々 `string | undefined` だが、`undefined` の枝で `panic` を呼んだ時点でそれ以降の処理は実行されない。だからこの行を通過した `x` の取りうる値の集合は `undefined` を取り除いた `string` だけになる、というロジックである。

## エラーの読み方

`never` がエラーに出てきたら、空集合の視点で原因を絞り込める。

- **「`X` は型 `never` に代入できません」**：その文脈で取りうる値の集合が空。網羅性チェックや絞り込みの漏れを疑う。
- **プロパティ型が `never` に推論される**：両立しない型のインターセクション(`string & number` など)を疑う。
- **ユニオン型が思ったより小さい**：条件型のフィルタで不要なメンバーが `never` に潰れている。`Exclude` や条件型の分配を疑う。

## おわりに

`never` 型を「決して起こらない値の型」というだけで覚えていると、ユニオンから消えたり、インターセクションを呑み込んだりする挙動はやや奇妙に映る。一方、**空集合**として捉え直してしまえば、これらはすべて素直な集合演算の結果になる。

- `never` はボトム型 = 空集合
- ユニオンに対しては単位元として振る舞い、消える(`A | never = A`)
- インターセクションに対しては吸収元として振る舞う(`A & never = never`)、また「重ならない型同士の交差」も空集合になって `never` に潰れる
- 条件型と組み合わせるとユニオンのフィルタになる(`Exclude` の仕組み)
- 自前の `MyEffect<A, E>` を作ってみると、`succeed`/`fail` を三項演算子で合成したときに `never` が消える挙動が、型統一と空集合の性質によって自然に説明できる
- エラーメッセージで `never` が顔を出したら「ここで取りうる値の集合は空なんだな」と読み替えると、原因にあたりがつけやすい

`never` を「空集合」と読み替えるだけで、挙動の予測もエラーの原因特定もしやすくなる。
