## メタ情報(title / description)が記事内容を正確に表している

frontmatter の `title` と `description` は、記事一覧 / 検索結果 / SNS 共有 / OGP カードなどで記事の**入口**になる。これらが内容を正確かつ具体的に表していないと、読者の期待と本文がずれ、記事が届くべき相手に届かない。タイトルと description、本文の主題が三者で一致しているかを確認する。

### Title

**Bad(曖昧 / 内容が読み取れない):**

```yaml
title: 〇〇について
```

```yaml
title: やってみた
```

**Good(主題と提供価値を表現):**

```yaml
title: 〇〇を集合論の視点から整理する
```

**チェック観点:**

- 記事の主題と、読者が得られるもの(価値)が短く伝わるか
- 「〜について」「〜してみた」のような無情報な締めくくりに頼っていないか
- 具体的な技術名・概念名を含み、検索結果から記事を見つけやすいか
- ファイル名のスラグと意味的に一致しているか
- 内容に対して**過大な約束**(誇張表現・煽り)になっていないか

### Description

`description` は SEO・記事一覧・OGP 等で使われる短い要約。記事の本筋を 1〜2 文で言い切る。

**Bad(曖昧 / 動機しか書かれていない):**

```yaml
description: 〇〇について書きました。
```

```yaml
description: 最近気になったことのメモ。
```

**Good(題材と結論を端的に):**

```yaml
description: 〇〇の挙動を集合論の視点から整理し、なぜ△△が発生するのかを説明する。
```

**チェック観点:**

- 記事のテーマ(題材)と結論(何が分かるか)を含んでいる
- title の言い換えに留まっていない(title + α の情報量があるか)
- である調(常体)で統一されている
- 100〜150 字程度が目安。長すぎる場合は枝葉を削る

### Title × Description × 本文の整合

メタ情報の品質は三者の**整合性**で決まる。タイトルが約束した主題を description が要約し、本文が実際に提供している、という三段の一致を確認する。

**チェック観点:**

- title が示す主題と description の要約が一致しているか
- description で予告している結論が本文中に実際に書かれているか
- タイトルで具体的なライブラリ名 / バージョン / 対象範囲を宣言した場合、本文がその範囲を扱っているか
- `tags` がタイトル・本文の主題と一致しているか(無関係なタグや、漏れているタグがないか)
