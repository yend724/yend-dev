## 機密情報の混入と危険な設定の紹介に注意する

技術記事はコード片・コマンド出力・スクリーンショット経由で機密情報が露出しやすい。また、検証目的の安全でない設定をそのまま紹介すると、読者が本番に持ち込んでしまうリスクがある。**機密情報の混入チェック**と、**危険な設定を紹介する場合の注意書き**を確認する。

### 機密情報の混入

**Bad(実値が露出):**

```ts
const client = createClient({
  apiKey: "sk-live-AbCdEfGhIjKlMnOpQrStUvWxYz1234567890",
})
```

```sh
$ curl -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..." https://api.example.com/me
```

**Good(プレースホルダ / 環境変数経由):**

```ts
const client = createClient({
  apiKey: process.env.SERVICE_API_KEY!,
})
```

```sh
$ curl -H "Authorization: Bearer $TOKEN" https://api.example.com/me
```

**チェック観点:**

- API キー / トークン / パスワード / DB 接続文字列の**実値が貼られていないか**
- メールアドレス / 電話番号 / 個人を特定できる情報が混入していないか
- 社内システムのホスト名・IP・内部 URL が露出していないか
- ログ出力例に**他者の個人情報や認証情報の断片**が残っていないか
- スクリーンショットに**メール画面 / ブックマーク / 通知 / メニューの社名**などが映り込んでいないか
- git のコミットログ・URL に**プライベートリポジトリの内部情報**が混じっていないか

### 危険な設定を紹介する場合の注意書き

検証や説明のためにセキュリティ的に弱い設定(`--insecure`、CORS 全許可、認証スキップ、平文パスワード保存 等)を紹介する場合、**「本番では使わないこと」を明記**する。読者が文脈を取り違えて本番に持ち込まないよう、サンプルコードのコメントや前後の文章で警告を添える。

**Bad(無警告で危険設定を紹介):**

```ts
app.use(cors({ origin: "*" }))
```

**Good(用途・リスク・代替を明示):**

```md
ローカル動作確認のため、CORS をすべて許可する。**本番ではアクセス元 origin を限定すること**。

```ts
// NOTE: 開発用のみ。本番では origin を明示的に指定する。
app.use(cors({ origin: "*" }))
```
```

**チェック観点:**

- 「サンプル / 検証 / 開発用」と「本番運用想定」のどちらの設定かが**読者に伝わるか**
- 危険な設定には**警告と代替案**が添えられているか
- 認証 / 認可 / 入力検証 / 暗号化に関する説明で、**読者が誤って脆弱な実装を量産するパターン**になっていないか
- 依存ライブラリのバージョンが**既知の脆弱性を含むもの**になっていないか
