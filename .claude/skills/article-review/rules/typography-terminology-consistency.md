## 用語・表記が記事内で一貫している

同じ概念に対しては同じ用語を使う。複数の言い方を混ぜると、読者は別の概念を指していると誤解しやすい。固有名詞、ライブラリ名、機能名、和訳と原語の使い分けも記事内で一貫させる。

**Bad(同じ概念に複数の表記が混在):**

```md
コンポーネントの状態を保持するには State フックを使う。

useState は React の主要なフックの 1 つである。

ステイトを更新するときは setState 関数を呼び出す。
```

→ `State フック` / `useState` / `ステイト` / `setState` が同じ対象を指しているように見えるが、表記がブレており混乱を招く。

**Good(用語と表記を統一):**

```md
コンポーネントの状態を保持するには `useState` フックを使う。

`useState` は React の主要なフックの 1 つで、状態の値と更新関数(慣例的に `setX` と命名)を返す。

状態を更新するときは更新関数を呼び出す。
```

**チェック観点:**

- **同じ概念には同じ用語**を使い、言い換えで読者を混乱させない
- **固有名詞・ライブラリ名・機能名の表記揺れがない**(例: `Next.js` / `nextjs` / `Next`、`TypeScript` / `Typescript` / `ts` の混在を避ける)
- **訳語と原語の使い分けが一貫している**(初出時に「サイドエフェクト(side effect)」のように原語を併記したら、その後は片方に統一する)
- **大文字 / 小文字** の使い分けが正しい(製品の正式表記に合わせる: `JavaScript` であって `Javascript` ではない、など)
- **inline code とそうでない表記** の使い分けが揃っている(関数名・型名・ファイル名は `` ` ` `` で囲む等)
- **記号・約物の表記揺れがない**(全角/半角の括弧、ダッシュとハイフンの混在など)

**確認手段:**

- 主要な固有名詞や用語を一度書き出し、それぞれを記事内検索して表記が揃っているか確認する
- 機械的に確認しにくい言い換えパターン(`配列` / `リスト`、`関数` / `メソッド` 等)は通読でチェック
