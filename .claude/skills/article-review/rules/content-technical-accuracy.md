## 記事内容が技術的に正確である

技術ブログの信頼性は内容の正確さに直結する。**公式仕様との一致**、**読者環境での再現可能性**、**事実と推測の区別**の3点を確認する。

### 公式仕様との一致

ライブラリ・フレームワーク・言語仕様・API などを題材にする場合、本文の主張・コード・型定義・挙動の説明が**執筆時点の公式ドキュメントの内容と一致しているか**を確認する。古いバージョンの仕様や、すでに deprecated になった API を現役のように紹介していないか点検する。

**Bad(古い仕様 / 出典不明):**

```md
このフレームワークでは旧 API の `oldMethod()` を使う。
```

**Good(現行仕様 + 出典を明示):**

```md
[Getting Started](https://example.com/docs/getting-started/)によれば、現行バージョンでは `newMethod()` を使う。
```

**チェック観点:**

- 紹介している API / フラグ / オプション名が現行の公式ドキュメントに存在するか
- バージョン依存の挙動を扱う場合、対象バージョンを明記しているか
- 主張・引用には出典(公式ドキュメント等)へのリンクが付いているか
- 引用文が原文と齟齬なく、最新の表記に追従しているか

### 読者環境での再現可能性

読者が同じ手順を試したときに動くことが、技術ブログの実用性を支える。**環境・バージョン・依存関係を明記**し、紹介するコード/コマンドが想定どおりに動作することを確認する。

**Bad(環境・バージョン不明):**

```md
以下のコマンドでビルドできる。

```sh
npm run build
```
```

**Good(環境前提を明示):**

```md
Node.js 22.0+ / pnpm 9 を前提とする。リポジトリのルートで以下を実行する。

```sh
pnpm install
pnpm build
```
```

**チェック観点:**

- OS / ランタイム / ライブラリの**バージョンが明記されているか**
- 必要な**依存関係・設定ファイル・環境変数**が省略されていないか
- 紹介するコードやコマンドを実際に実行して動作を確認したか
- セットアップ手順を最初から最後まで通したとき、行間で詰まる箇所がないか

### 事実と推測の区別

「公式仕様として定められていること」「自分が検証した結果」「自分の解釈・推測・経験則」は、読者にとって信頼度が違う。混在させると、推測が仕様であるかのように伝わってしまう。

**Bad(推測を仕様のように書く):**

```md
このオプションを `true` にするとパフォーマンスが2倍になる。
```

**Good(事実と推測を区別):**

```md
[公式ドキュメント](https://example.com/docs/options/)によれば、このオプションは内部的にキャッシュ層を有効化する。
手元の環境で計測した範囲では、本記事のサンプル相当のワークロードで2倍程度の速度向上が見られた。性能効果は環境依存のため、実際のワークロードで計測することを推奨する。
```

**チェック観点:**

- 「公式仕様 / 規格」「実測値・検証結果」「個人的な解釈・推測」が読み分けられるか
- 性能・セキュリティ・保守性に関する主張に**根拠(出典・計測条件・実例)が添えられているか**
- 「常に〜」「必ず〜」「推奨される」などの**断定的な表現**が、自分の経験則を仕様のように述べていないか
- 検証していない可能性に「〜のはず」「〜と思われる」など曖昧さを保つ表現が使われているか

### 確認手段

- 該当ライブラリの**最新の公式ドキュメント**を参照(必要に応じて context7 などのツールで取得)
- 公式リリースノート / changelog を確認
- 書いたコードがそのバージョンで実際に動作するか試す
- 自分が手を動かさずに書いた箇所がないかを意識的に振り返る
