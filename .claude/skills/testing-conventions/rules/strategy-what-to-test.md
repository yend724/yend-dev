## テストすべきもの・スキップしてよいものの判断基準

すべてのコードにテストを書く必要はない。テストの価値は4つの柱で決まる。これらのバランスが良い箇所にテストを集中させる。

### テストの価値を判断する4つの柱

- **リグレッション防止** — そのテストはバグを見つけられるか？複雑なロジックほど高い
- **リファクタリング耐性** — 内部を変更してもテストが壊れないか？振る舞いベースのテストほど高い
- **高速なフィードバック** — テストはすぐに結果を返すか？単体テストほど高い
- **保守性** — テストは読みやすく変更しやすいか？簡潔な構造ほど高い

リファクタリング耐性は妥協してはならない。これが低いテスト（偽陽性が多いテスト）はチームのテストへの信頼を損なう。

### テストすべきもの

| 対象 | テスト種別 | 理由 |
|------|-----------|------|
| ビジネスロジック / ドメインロジック | Vitest 単体テスト | 複雑さが高く、バグの影響が大きい |
| ユーザーインタラクションを持つコンポーネント | Vitest + Testing Library | 条件分岐やイベント処理のあるUIは壊れやすい |
| フォームのバリデーション | Vitest 単体テスト | エッジケースが多い |
| 主要なユーザージャーニー（認証、購入等） | Playwright E2E | ビジネスインパクトが大きい |
| 複雑な状態遷移 | Vitest 単体テスト | 状態の組み合わせが多く手動テストでは漏れる |

### スキップしてよいもの

| 対象 | 理由 |
|------|------|
| 表示のみの Server Component | ロジックがなく壊れにくい。E2E でカバーされる |
| 外部ライブラリの挙動 | ライブラリ側でテスト済み |
| 単純な props の受け渡し | TypeScript の型チェックで十分 |
| フレームワークの規約に従っただけのコード（layout.tsx 等） | フレームワークが保証する |
| 複雑さが低く協力オブジェクトも少ないコード | テストの価値が低い |

### テストを追加する判断

- バグが発生した箇所 → 再発防止のテストを追加
- コードレビューで「ここ壊れそう」と感じた箇所 → テストを追加
- 「テストがないと不安でリファクタリングできない」箇所 → テストを追加
